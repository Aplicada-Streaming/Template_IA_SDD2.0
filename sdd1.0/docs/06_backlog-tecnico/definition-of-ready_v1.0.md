# 1. Propósito

Definir cuándo un ítem del backlog está listo para ser incluido en un Sprint Planning. Un ítem que no cumple estos criterios no debe ser comprometido en el sprint.

---

# 2. Criterios de Ready

Un ítem del backlog se considera **Ready** cuando cumple TODOS los siguientes criterios:

- [ ] Descripción clara y comprensible por todo el equipo
- [ ] Criterios de aceptación definidos (preferiblemente en formato Given/When/Then)
- [ ] Caso de uso asociado identificado (CU-XX) cuando aplica
- [ ] Dependencias identificadas y resueltas (o planificadas)
- [ ] Estimación de story points realizada por el equipo
- [ ] Tareas técnicas descompuestas (BT-XXX) cuando aplica
- [ ] Sin bloqueos conocidos sin plan de mitigación
- [ ] Datos de prueba identificados o disponibles

---

# 3. Criterios Adicionales para Historias de Usuario

Adicionalmente, las historias de usuario deben cumplir:

- [ ] Escrita en formato "Como [rol], quiero [acción], para [beneficio]"
- [ ] Product Owner ha validado la necesidad
- [ ] El equipo entiende el valor de negocio

---

# 4. Proceso de Refinamiento

Los ítems se refinan en sesiones de **Backlog Refinement** antes del Sprint Planning. Durante estas sesiones:

- Se revisan los ítems candidatos para el próximo sprint.
- Se aclaran dudas de negocio y técnicas.
- Se estiman story points mediante Planning Poker u otra técnica acordada.
- Se descomponen historias grandes en ítems más pequeños si es necesario.
- Se verifican los criterios de Ready para cada ítem.

---

# 5. Notas

- Si un ítem no cumple el DoR, **no debe incluirse en el sprint commitment**.
- El Product Owner es responsable de asegurar que los ítems estén refinados con suficiente anticipación.
- El equipo puede devolver ítems al backlog durante el Sprint Planning si no cumplen el DoR.
- Este documento complementa el [Definition of Done](../08_calidad_y_pruebas/definition-of-done_v1.0.md).
